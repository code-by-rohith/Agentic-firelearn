# Project Functions in Plain Language (120-150 Words Each)

## `agentic_rag/src/rag_tools.py`

### `SimpleVectorIndex.__init__(self, documents)`
This function prepares a searchable knowledge shelf from the document list you give it. It is used at startup, before anyone asks questions, so the system is ready to look things up quickly. It receives a collection of text items, each with a name and full content, and stores them in memory. After storing them, it immediately organizes all words so later lookups are faster and more accurate. The result is a ready helper that other parts can ask for the best matching documents. Without this step, every new question would force the system to re-read and re-organize everything from the beginning, which would be slow and repetitive. In simple terms, this is the setup job that turns raw reading material into a prepared reference library.

### `SimpleVectorIndex._tokenize(text)`
This function turns a piece of writing into a clean list of simple words that can be compared fairly. It is used whenever the system reads either document text or a user question, because both need the same cleaning style. It takes one text block, makes all letters consistent, and removes punctuation noise so only useful word parts remain. The result is a word list that can be measured and matched against other word lists. This makes lookup quality better, because “Rules,” “rules,” and similar forms are treated like the same idea. In everyday terms, this function is like cleaning and sorting paper notes before filing them. It helps the system focus on meaning words, not visual formatting differences that should not change the answer quality.

### `SimpleVectorIndex._build(self)`
This function studies all stored documents and prepares the scoring values used to decide which document best matches a question. It runs during setup, right after documents are loaded, and reads every word list from every source file. It counts how common each word is across the whole collection and also how strongly each word describes each single document. The result is a full set of readiness data that later lookup steps can use immediately. In simple terms, it labels which words are especially informative and which are too common to be very helpful. This gives better ranking when someone asks a question. Instead of treating every word equally, it gives more attention to words that truly separate one topic from another and better identifies useful source material.

### `SimpleVectorIndex._embed_query(self, query)`
This function converts a user question into the same comparable form used for stored documents. It is used each time a question arrives, just before document matching starts. It takes the question text, breaks it into cleaned words, and builds a weighted summary of those words so it can be compared fairly with document summaries. It also returns a size number that helps keep comparisons balanced. The result is a prepared question profile that the ranking step can use directly. In simple terms, this function rewrites the question into the system’s matching format. If the question has no useful words, it returns an empty result so the system can avoid useless matching. That prevents confusing outputs and keeps search behavior steady for real user questions.

### `SimpleVectorIndex._dot(a, b)`
This function calculates how much two prepared word summaries overlap. It is used during matching, when the system compares one question summary against one document summary. It takes two sets of word values, checks the words they share, and combines those shared values into one overlap number. The result is a single score that says how strongly the two texts are related at the word level. In simple terms, this is the “how similar are these two” calculator. A higher number means stronger connection. This number is not shown directly to most users, but it is important because it helps choose which documents come first in results. It allows the system to compare many sources in a consistent, measurable way instead of guessing relevance.

### `SimpleVectorIndex.search(self, query, top_k=3)`
This function finds the best matching documents for a question and returns only the top few results. It is used whenever the system needs evidence text before composing an answer. It takes the question and the desired number of results, compares the question against every stored document, gives each document a relevance score, and sorts the list from strongest to weakest. The result is a short ranked set of matches, usually the most useful pieces for answering. In simple terms, this is the main lookup step. It prevents information overload by returning only the highest matches rather than all files. That makes answers more focused and easier to trust, because the chosen sources are the ones most closely tied to what the person asked.

### `_load_documents(data_dir)`
This function reads all text files from a folder and turns them into the knowledge items used by the system. It is used at startup, before searching can happen, to collect source material from disk. It takes one folder location, opens each text file in that folder, captures the file name as a source label, and stores the file content. The result is a list of document entries ready for indexing and matching. If no files are found, it stops and raises an error because there would be nothing useful to answer from. In simple terms, this is the content loading step that fills the library shelves. It ensures the assistant has real written references before anyone depends on it for information.

### `plan_query(user_query)`
This function creates a cleaned first version of the user question and returns it with a short search plan note. It is used as the opening step before document lookup starts. It takes the user’s text, removes extra spaces around it, and places that cleaned text into a small result object. The result includes the planned question and a plain sentence about the first attempt strategy. In simple terms, this function makes sure every question starts from a tidy format instead of inconsistent typing styles. It does not change the meaning deeply, but it provides a stable first step that other steps can rely on. That consistency helps the system produce repeatable behavior when multiple people ask questions in different writing styles.

### `refine_query_if_needed(user_query)`
This function expands a question with related terms when the original wording may be too short to find good matches. It is used as a second attempt helper after weak or empty first results. It takes the question text, checks for known shortcut words like auth, db, rules, price, and hosting, then adds fuller related phrases. The result is a new question string plus a true or false marker showing whether expansion was applied. In simple terms, it gives the lookup step more clues. If a person asks in very brief language, this function adds common related wording found in source documents. That improves the chance of finding relevant evidence and reduces “no useful match found” outcomes during normal question answering.

### `retrieve_context(query, top_k=3)`
This function retrieves the best evidence snippets for a question and packages them in a clear, compact format. It is used after planning, when the system needs real source text to support an answer. It takes a question and a desired result count, keeps that count within a safe range, runs document matching, and then prepares each hit with source name, rounded score, and a short opening snippet. The result is a simple context bundle containing the question, number of matches, and snippet list. In simple terms, this function collects proof text. It gives answer-writing steps concrete material to cite, rather than relying on memory-style guessing. That makes final responses more grounded, traceable, and easier for readers to verify against known source files.

### `list_knowledge_sources()`
This function lists all source documents currently loaded and tells how many there are. It is used when someone wants to confirm what knowledge is available before asking detailed questions. It takes no input and simply reads the in-memory document list created at startup. The result is a compact summary with source names and total count. In simple terms, this is the “what is on the shelf” check. It helps teams quickly see whether the right files were loaded and whether any expected topic file is missing. That matters when answers seem incomplete, because the issue may be missing content rather than bad matching. This function gives a fast visibility check without requiring people to inspect folders manually or run full question searches.

## `agentic_rag/src/agentic_rag.py`

### `SimpleVectorIndex.__init__(self, documents)`
This function sets up the searchable document collection used by the interactive question mode in this file. It is used when the assistant object is first created, before the conversation loop begins. It takes a list of document entries, stores them, creates empty storage for scoring details, and then prepares all matching data right away. The result is a search helper that can immediately answer “find best matching text” calls. In simple terms, this is the preparation step that turns loaded documents into a ready lookup engine. Doing this once at the beginning keeps later answers faster because the assistant does not repeat the same heavy preparation for each question. It is the same idea as organizing a filing cabinet before staff start looking for specific records.

### `SimpleVectorIndex._tokenize(text)`
This function converts a text block into a clean list of words so comparison is fair and consistent. It is used both for document content and for user questions in this module’s interactive mode. It takes one text input, makes it lowercase, removes punctuation-style noise, and keeps only alphanumeric word parts. The result is a simple word list that matching steps can compare directly. In plain terms, it standardizes writing so small style differences do not hide real topic similarity. For example, words with different letter case still count as the same word. This function helps the assistant treat all text equally and keeps lookup results stable across different writing styles. It is a basic cleaning step that makes every later matching step more dependable.

### `SimpleVectorIndex._build(self)`
This function builds the internal scoring map from all loaded documents so questions can be matched accurately. It is used during setup and reads every document already stored in the object. It counts which words appear in how many documents and builds weighted summaries for each document, then stores summary sizes used in scoring. The result is a prepared ranking base that later matching calls use directly. In simple terms, this function decides which words are especially useful for distinguishing topics and stores that information for fast reuse. By preparing these values once, the system avoids repeated heavy analysis for every question. That keeps interactive questioning practical and makes rankings more reliable because the same prepared rules are reused consistently across all searches.

### `SimpleVectorIndex._embed_query(self, query)`
This function transforms a user question into the same weighted summary style used for stored documents. It is used each time someone asks a question in the interactive mode, just before matching starts. It takes the question text, cleans and splits it into words, applies the learned word importance values, and calculates a size number for fair comparison. The result is a pair containing the weighted question summary and its size. In simple terms, this gives the system a standardized view of the question so it can compare “like with like.” If there are no meaningful words, it returns an empty summary so matching can safely skip poor comparisons. This keeps rankings clearer and avoids misleading results caused by empty or nearly empty questions.

### `SimpleVectorIndex._dot(a, b)`
This function measures overlap between two weighted word summaries and returns one numeric value. It is used during ranking when one question summary is compared against one document summary. It takes two summary sets, checks shared words, multiplies shared values, and adds them together. The result is a single overlap score used to calculate final relevance. In plain terms, it answers: “How much do these two texts line up?” The function also compares smaller sets first to reduce unnecessary work. While users do not see this number directly, it strongly affects which sources are ranked first. This function gives the ranking process a clear and repeatable mathematical core so results are consistent and based on measurable similarity instead of loose text guessing.

### `SimpleVectorIndex.search(self, query, top_k=3)`
This function compares a question with all documents and returns the strongest matches. It is used in the interactive assistant whenever a person asks something and source evidence is needed. It takes two inputs: the question text and how many top results should be returned. It creates a question summary, compares it with each document summary, computes relevance scores, removes non-useful matches, sorts by score, and returns the best few. The result is an ordered list of matched document entries with score values. In simple terms, this is the ranking gate that chooses which sources are worth showing or using in the final answer. It keeps output focused and practical by limiting results to the most relevant evidence rather than returning an unfiltered pile of text.

### `AgenticRAG.__init__(self, data_dir)`
This function creates a ready question-answer assistant using documents from a chosen folder. It is used once when the assistant starts. It takes the folder location, loads all source texts from that location, and builds a searchable collection from those sources. The result is a complete assistant object with both content and lookup ability prepared. In simple terms, this is the startup assembly step: gather knowledge first, then prepare matching so questions can be answered immediately. If the folder is empty, startup fails early instead of pretending everything is fine. That early stop prevents confusing behavior later. This function makes sure the assistant begins in a usable state, with known documents loaded and searchable before anyone enters a question in the interactive session.

### `AgenticRAG._load_documents(data_dir)`
This function reads text files from a folder and converts them into named source entries for the assistant. It is used during assistant startup before any question can be answered. It takes one folder location, scans for text files, reads each file’s content, and stores both the source name and full text. The result is a document list that later matching steps can index and rank. If no text files are present, it raises an error so missing content is obvious immediately. In simple terms, this function fills the assistant’s knowledge library from disk. It ensures there is real material to draw from, and it keeps source naming consistent so later answers can mention where each supporting snippet came from when building grounded, understandable responses.

### `AgenticRAG._plan_query(user_query)`
This function creates a cleaned question string by removing extra spaces around the user text. It is used as the first step whenever the interactive assistant processes a new question. It takes the raw question exactly as typed, trims outer whitespace, and returns the cleaned version. The result is simple but important: a consistent starting question for matching and possible retry steps. In plain terms, this function does basic text tidying so avoidable formatting noise does not affect lookup behavior. It does not rewrite meaning or add new terms; it keeps the person’s wording intact. Having this dedicated step also makes future improvements easier, because richer planning rules can be added here later without changing the rest of the answer path.

### `AgenticRAG._expand_query_if_needed(original_query, first_pass)`
This function decides whether to broaden the question after the first matching attempt and returns the final search phrase. It is used between first and second lookup attempts in the interactive assistant. It takes the original question and the first match list. If first results exist, it keeps the original wording. If no results exist, it checks for known short terms and appends related words like authentication or security rules language. The result is either unchanged text or an expanded question designed for a stronger second try. In plain terms, this function gives the system a fallback when original wording is too narrow. It helps recover useful matches without forcing the person to manually rewrite the question and retry repeatedly on their own.

### `AgenticRAG._compose_answer(query, contexts)`
This function turns selected evidence snippets into the final answer text shown to the person. It is used after matching finishes, once the top source entries are known. It takes the original question and the list of matched contexts. If there are no contexts, it returns a clear message saying relevant material was not found. If contexts exist, it builds a grounded summary, includes short snippets from each source, and lists source names for reference. The result is one readable answer string. In simple terms, this is the final writing step that converts raw match data into a human-friendly response. It keeps answers tied to source material instead of free-form guessing and makes it easier for readers to understand where important statements came from.

### `AgenticRAG.ask(self, user_query, top_k=3)`
This function runs the full question path from user text to final grounded answer in one call. It is used whenever someone asks a question in the interactive mode. It takes the question and desired top match count, cleans the question, performs first matching, optionally expands wording if no results appear, performs second matching when needed, and finally composes a readable answer from the selected evidence. The result is a complete response string ready to display. In plain terms, this is the main “answer this question” function of the class. It combines preparation, lookup, retry logic, and final writing so callers do not need to manage separate steps. That keeps usage simple and ensures each question is treated with the same consistent logic.

### `run_cli()`
This function starts a command-line conversation so a person can ask many questions and get answers immediately. It is used when this file is run directly as a program. It takes no input at start, loads the assistant with local knowledge files, prints guidance text, and enters a loop waiting for typed questions. For each non-empty question, it calls the answer function and prints the result. If the person types exit or quit, it ends the session gracefully. The result is an interactive text experience for testing and everyday use. In plain terms, this function is the front door for manual use: open the assistant, type questions, and see grounded answers. It offers a simple way for new team members to explore behavior without extra tools.

## `agentic_rag/main.py`

### `demo(query)`
This function runs a single demonstration of the planning and matching steps using one question. It is used for quick checks when someone wants to see intermediate outputs without starting the full interactive loop. It takes one question text, creates the first planned version, performs initial context matching, and if no matches are found, expands the question and matches again. It then prints both plan data and final matching results to the console. The result is not a polished end-user answer but a transparent view of what the lookup steps produced. In plain terms, this is a one-shot test run that shows how the system reacts to a sample question. It helps people confirm whether source matching and retry logic are behaving as expected.

## `metaagent/remote_a2a/time_agent/agent.py`

### `get_current_time()`
This function reads the current date and time from the machine clock and returns it in a clear text format. It is used whenever the time assistant is asked for current time information. It takes no input values, simply checks “now” at the moment it is called, formats that value as year-month-day and hour-minute-second, and returns it inside a small result object. The result is a precise current timestamp that other parts can show directly to users. In plain terms, this is the clock lookup task. It provides factual time output instead of estimated or remembered time statements. That makes time answers dependable for practical use, especially when people need an exact current reading rather than a general conversation-style response.
