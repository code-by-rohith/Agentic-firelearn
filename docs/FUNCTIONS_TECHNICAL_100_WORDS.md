# Function Documentation (Direct Technical Style)

This document lists all source-defined functions and methods in the project and explains what each one does in a direct, practical tone.

## `agentic_rag/src/rag_tools.py`

### `SimpleVectorIndex.__init__(self, documents: List[Document]) -> None`

Initializes the in-memory retrieval index from a list of documents and precomputes all structures needed for search.  
This constructor takes `documents`, stores them on the instance, initializes empty dictionaries for document vectors, document norms, and inverse document frequency values, and immediately calls `_build()` to populate those structures. It does not return a value; its output is internal state that later methods depend on. It is used at module load time through `_INDEX = SimpleVectorIndex(_DOCS)`, so search is ready before any query arrives. If `documents` is empty, the issue is typically caught earlier by `_load_documents`. This method exists to front-load indexing work and keep per-query search operations focused on scoring rather than repeated corpus preprocessing.

### `SimpleVectorIndex._tokenize(text: str) -> List[str]`

Converts raw text into normalized alphanumeric tokens for indexing and query matching.  
This static method takes one string, lowercases it, and extracts word-like tokens with the regex pattern `[a-zA-Z0-9]+`. It returns a list of tokens and does not mutate object state. It is called by both `_build()` and `_embed_query()`, so document text and query text are transformed by the exact same rule set. That consistency is critical because scoring compares token keys directly. The method is intentionally strict and simple: punctuation is ignored, and only alphanumeric sequences are kept. It exists to provide deterministic tokenization across all retrieval operations, reducing mismatch caused by casing or punctuation differences in user queries and knowledge files.

### `SimpleVectorIndex._build(self) -> None`

Builds document-level weighted vectors and norms used for cosine similarity search.  
This method reads `self.documents`, tokenizes each document, computes document frequency counts, derives IDF values, then computes per-document TF-IDF-like vectors and Euclidean norms. It stores results into `self.idf`, `self.doc_vectors`, and `self.doc_norms`. It does not return a value; it prepares internal data for `search()`. The method is used during index initialization and assumes documents are already loaded in memory. Its scoring formula gives less weight to common terms and more weight to discriminative terms. This method exists so query-time work stays small: search can reuse cached vectors and norms rather than recalculating corpus statistics for every request.

### `SimpleVectorIndex._embed_query(self, query: str) -> Tuple[Dict[str, float], float]`

Transforms a user query string into the same weighted vector space used by document vectors.  
This method takes `query`, tokenizes it, computes term frequencies, multiplies by existing IDF values, and returns a tuple of `(query_vector, query_norm)`. If the query yields no tokens, it returns `({}, 0.0)`. It reads `self.idf` but does not modify index state. `search()` calls it first and uses the returned norm to guard against divide-by-zero and empty-query scoring. Unknown tokens effectively receive zero weight because they are not present in IDF. This method exists to standardize query representation so similarity scores are comparable to document vectors generated by `_build()`, enabling predictable ranking behavior across different query phrasings.

### `SimpleVectorIndex._dot(a: Dict[str, float], b: Dict[str, float]) -> float`

Computes sparse vector dot product between two token-weight dictionaries.  
This static helper takes two vectors `a` and `b`, swaps them when `a` is larger, and iterates the smaller dictionary to reduce lookups. It returns a single float equal to the sum of overlapping token products. The method has no side effects and no external dependencies beyond dictionary operations. `search()` uses it as the numerator in cosine similarity calculations. The optimization of iterating the smaller map keeps scoring efficient when vector sizes differ significantly, which is common in sparse text representations. This method exists to provide a compact, reusable, and performant primitive for similarity math without duplicating dot-product logic in multiple places.

### `SimpleVectorIndex.search(self, query: str, top_k: int = 3) -> List[RetrievalResult]`

Scores all indexed documents against a query and returns the top matches.  
This method takes `query` and `top_k`, embeds the query via `_embed_query()`, and exits early with `[]` when query norm is zero. It then iterates every document, retrieves each document vector and norm, computes cosine similarity using `_dot()`, filters out non-positive scores, sorts by score descending, and returns up to `top_k` `RetrievalResult` objects. It reads precomputed index state and does not alter corpus data. This method is used by `retrieve_context()` and effectively drives evidence selection. It exists to produce ranked relevance results in a single call, keeping caller code focused on formatting and response logic rather than ranking implementation details.

### `_load_documents(data_dir: Path) -> List[Document]`

Loads `.txt` files from disk into `Document` objects for indexing.  
This function takes a folder path `data_dir`, reads all matching text files in sorted order, and creates a list of `Document(doc_id, text)` entries using the file stem as `doc_id`. It returns the document list and raises `ValueError` if no `.txt` files are found. It reads from local storage and performs UTF-8 decoding. The function is called at module import to build `_DOCS`, which then feeds index construction. It exists to isolate file-loading logic from retrieval logic and to fail fast when knowledge content is missing. That early validation prevents silent startup with an empty corpus and avoids confusing zero-result behavior at query time.

### `plan_query(user_query: str) -> dict`

Normalizes incoming query text and returns a structured first-pass retrieval plan.  
This function takes `user_query`, strips surrounding whitespace, and returns a dictionary with `planned_query` and a static strategy description. It does not access external services or global state and does not modify persistent data. It is used as the first step in demo and agent tool flows so retrieval input follows a consistent shape. The return format is intentionally explicit, which allows callers to inspect plan output and potentially log or display it. This function exists as a stable planning interface: current behavior is simple normalization, but the structure supports future expansion to richer planning logic without changing the consumers that expect a dictionary contract.

### `refine_query_if_needed(user_query: str) -> dict`

Expands query text with predefined domain terms to improve second-pass retrieval.  
This function takes `user_query`, checks tokens against a fixed expansion map (for terms like `auth`, `db`, `rules`, `price`, `hosting`), appends related phrases when matches exist, and returns `{"refined_query": ..., "expanded": ...}`. It reads only local mapping data and input text; no network or file access is involved. It is typically called when first retrieval returns weak or zero results. The boolean `expanded` explicitly indicates whether modifications were applied. This function exists to increase recall for short or shorthand queries by injecting known domain vocabulary, reducing false negatives when users omit terms that appear in source documents.

### `retrieve_context(query: str, top_k: int = 3) -> dict`

Runs retrieval and returns a compact, display-ready context payload with source snippets.  
This function takes `query` and `top_k`, clamps `top_k` into `[1, 8]`, calls `_INDEX.search()`, and transforms each hit into `{source, score, snippet}` where snippet is the first 36 words of document text. It returns `{"query": ..., "count": ..., "contexts": [...]}`. It reads from the module-level index built at import and does not mutate source files. This function is used directly by the ADK agent tools and demo scripts as the main evidence-fetching call. It exists to bridge low-level ranking output and answer-generation needs by providing concise, structured retrieval results that are easy to cite and inspect.

### `list_knowledge_sources() -> dict`

Returns the names and count of currently indexed knowledge sources.  
This function takes no arguments and returns `{"sources": [...], "count": ...}` derived from `_DOCS`. It reads in-memory document metadata loaded during module initialization and performs no heavy computation. It is used as an inspection utility so callers can verify what content is available for retrieval before or during question handling. The output is deterministic for a given loaded corpus. This function exists to provide lightweight observability into corpus coverage, helping diagnose issues such as missing files or unexpected source sets without requiring direct file-system checks or full retrieval calls.

## `agentic_rag/src/agentic_rag.py`

### `SimpleVectorIndex.__init__(self, documents: List[Document]) -> None`

Initializes the CLI RAG index with document vectors and scoring metadata.  
This constructor receives a list of `Document` objects, stores them, allocates dictionaries for vectors, norms, and IDF values, and triggers `_build()` immediately. It returns `None` and sets internal state used by `search()`. The behavior mirrors the index class in `rag_tools.py`, but it is scoped to the class-based CLI implementation. It is called from `AgenticRAG.__init__` after document loading. This method exists to ensure the assistant instance is fully searchable right after creation. By building index state once at startup, repeated calls to `ask()` can focus on retrieval and answer composition without recomputing corpus statistics each time.

### `SimpleVectorIndex._tokenize(text: str) -> List[str]`

Splits text into normalized alphanumeric tokens for consistent query/document comparison.  
This static method takes a raw text string, lowercases it, and extracts tokens using regex. It returns a list of token strings and has no side effects. It is called by `_build()` for document processing and `_embed_query()` for query processing, ensuring both paths use identical token boundaries. This function exists to avoid token inconsistency that would otherwise reduce retrieval quality. Its simple pattern intentionally ignores punctuation and focuses on alphanumeric terms, which keeps indexing behavior predictable and easy to reason about when debugging relevance results in the CLI assistant flow.

### `SimpleVectorIndex._build(self) -> None`

Precomputes corpus term statistics and per-document weighted vectors for ranking.  
This method processes `self.documents`, computes document frequency counts, calculates IDF values, derives TF-IDF-like vectors for each document, and stores vector norms. It writes results into `self.idf`, `self.doc_vectors`, and `self.doc_norms`, returning no explicit value. It is invoked during index initialization and assumed to run before any call to `search()`. This method exists to separate one-time corpus preparation from query-time scoring. The precomputation design keeps retrieval latency lower during interactive sessions because each query reuses cached vector data instead of recalculating all document term statistics.

### `SimpleVectorIndex._embed_query(self, query: str) -> Tuple[Dict[str, float], float]`

Creates a weighted sparse vector representation of the query for similarity scoring.  
This method takes `query`, tokenizes it, computes normalized term frequencies, multiplies by existing IDF values, and returns `(vector, norm)`. For empty token sets, it returns an empty vector and zero norm. It reads index state from `self.idf` and does not modify stored document vectors. `search()` depends on this output to compute cosine similarity safely. This method exists to keep query preparation logic isolated and consistent with document vector construction rules. That separation simplifies maintenance and makes scoring behavior easier to validate when tuning retrieval outcomes or diagnosing unexpected ranking results.

### `SimpleVectorIndex._dot(a: Dict[str, float], b: Dict[str, float]) -> float`

Calculates dot product for sparse token vectors during similarity evaluation.  
This helper takes two token-weight dictionaries, iterates the smaller one for efficiency, and sums products for overlapping keys. It returns one float and has no side effects. `search()` uses it repeatedly across query-document comparisons. The function exists as a focused primitive to keep similarity math concise and reusable, rather than embedding repeated dictionary intersection logic inside the search loop. Its efficiency matters because it runs once per document candidate and can dominate runtime in larger corpora. The behavior is deterministic given inputs, which also helps reproducibility when inspecting ranking calculations.

### `SimpleVectorIndex.search(self, query: str, top_k: int = 3) -> List[RetrievalResult]`

Ranks documents by cosine similarity against the query and returns the top results.  
This method takes `query` and `top_k`, builds query vector data, skips scoring if query norm is zero, computes similarity with each document vector, filters positive scores, sorts descending, and returns up to `top_k` entries. The return type is `List[RetrievalResult]`. It reads precomputed vectors/norms from instance state and does not alter corpus content. `AgenticRAG.ask()` calls this method for first-pass retrieval and optional second-pass retrieval after query expansion. The method exists to centralize ranking behavior in one place, making caller code simpler and ensuring all retrieval calls follow the same scoring logic.

### `AgenticRAG.__init__(self, data_dir: Path) -> None`

Creates a ready-to-query RAG assistant by loading documents and building an index.  
This constructor takes `data_dir`, calls `_load_documents()` to read text sources, stores them in `self.documents`, and initializes `self.index` with `SimpleVectorIndex`. It returns no value; the output is a fully configured assistant instance. It is used by `run_cli()` at startup and is the main entry point for class-based RAG setup. This method exists to encapsulate initialization in one call so callers do not need to manually load files and build the index in separate steps. It also makes startup failures explicit, such as missing data files, before interactive queries begin.

### `AgenticRAG._load_documents(data_dir: Path) -> List[Document]`

Reads local text files and converts them into document records for retrieval.  
This static method takes a directory path, scans for `.txt` files, reads each file as UTF-8, and returns a list of `Document` objects with file stem IDs. It raises `ValueError` when no documents are found. It is called by `AgenticRAG.__init__` and does not depend on instance state. The method exists to keep corpus loading logic explicit and reusable within the class. Sorting file paths ensures deterministic ordering, which helps debugging and predictable output when comparing runs. The returned documents directly feed index construction and define the knowledge base available to `ask()`.

### `AgenticRAG._plan_query(user_query: str) -> str`

Normalizes the user query string before retrieval by trimming outer whitespace.  
This static method takes `user_query`, applies `strip()`, and returns the cleaned string. It does not read external resources, mutate object state, or perform advanced rewriting. `ask()` calls it as the first step, making it the canonical point where raw input becomes retrieval input. The method exists to keep a clear planning hook in the class pipeline: current behavior is minimal, but this location can be expanded later without changing `ask()` structure. Even in its current simple form, it prevents formatting noise from propagating into retrieval calls and keeps query handling consistent across requests.

### `AgenticRAG._expand_query_if_needed(original_query: str, first_pass: List[RetrievalResult]) -> str`

Conditionally appends predefined domain terms when first-pass retrieval returns no hits.  
This static method takes the original query and first retrieval result list. If `first_pass` is non-empty, it returns the original query unchanged. Otherwise, it checks query words against a small expansion dictionary and appends related terms when matches exist. It returns the resulting query string. `ask()` uses this return value to decide whether to rerun retrieval. The method exists to improve recall for shorthand or underspecified queries without introducing model calls. Its behavior is deterministic and transparent, which makes it straightforward to debug and tune by editing the expansion map.

### `AgenticRAG._compose_answer(query: str, contexts: List[RetrievalResult]) -> str`

Formats retrieval results into a single grounded answer string with source references.  
This static method takes the original user query and a list of retrieval hits. If hits are empty, it returns a fallback message indicating no relevant context was found. If hits exist, it extracts short snippets, identifies the top source, composes a multi-line response, and includes citations using source IDs. It returns one string for display. `ask()` delegates final response formatting to this method. It exists to separate presentation logic from retrieval logic, keeping `ask()` focused on decision flow while ensuring responses consistently include evidence details that reflect the retrieved contexts.

### `AgenticRAG.ask(self, user_query: str, top_k: int = 3) -> str`

Executes the full RAG cycle for one question and returns the final answer text.  
This method takes `user_query` and `top_k`, runs `_plan_query()`, performs first retrieval via `self.index.search()`, optionally expands query with `_expand_query_if_needed()` when needed, optionally performs second retrieval, then formats output with `_compose_answer()`. It returns a single string response. It reads instance index and helper methods; it does not modify source files. This method is called from CLI interaction and represents the primary question-answer API in the class-based module. It exists to keep end-to-end behavior in one method so callers can request an answer without manually orchestrating planning, retrieval, retry logic, and formatting.

### `run_cli() -> None`

Starts an interactive command-line loop that repeatedly accepts questions and prints answers.  
This function builds `AgenticRAG` using the local `data` directory, prints a startup message, reads user input in a loop, handles exit commands (`exit`, `quit`), skips empty inputs, and prints `rag.ask(query)` for valid questions. It returns `None` when the loop ends. It is used when running `agentic_rag/src/agentic_rag.py` directly. This function exists as a local demo and manual testing entry point, allowing quick validation of retrieval behavior without integrating external interfaces. It provides a straightforward way to inspect how query planning, retrieval, optional expansion, and answer composition behave across different prompts.

## `agentic_rag/main.py`

### `demo(query: str) -> None`

Runs a single demonstration of plan-retrieve-refine logic and prints intermediate results.  
This function takes one input `query`, calls `plan_query()`, executes first retrieval with `retrieve_context()`, and if no results are found, calls `refine_query_if_needed()` followed by second retrieval. It chooses either first or second retrieval as final demo output, then prints plan and retrieval dictionaries. It does not return a value. It is used when running `agentic_rag/main.py` directly, including the default sample query under `__main__`. This function exists for quick inspection of tool behavior and output shape, making it easy to validate retrieval steps without running the full ADK agent setup or interactive CLI loop.

## `metaagent/remote_a2a/time_agent/agent.py`

### `get_current_time() -> dict`

Reads the current system date-time and returns it in a fixed string format.  
This function takes no arguments, calls `datetime.now()`, formats it as `%Y-%m-%d %H:%M:%S`, and returns `{"current_time": formatted_value}`. It has no side effects besides reading the current clock value at call time. It is registered as a tool on the time agent and can be invoked through the remote A2A interface created in the same module. This function exists to provide deterministic, tool-based time responses instead of relying on model text generation for current time. The output structure is stable and machine-readable, which makes it easy for calling agents to include or transform the value in user responses.
